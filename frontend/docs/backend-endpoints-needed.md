## 12. Демо-режим: auth-гейты и создание объявлений (frontend-заметка, не backend)

**Задача:** Финальная приёмка (2026-07-05).
**Статус:** `Frontend-only` — исправлено в этой приёмке, backend не требуется.
**Контекст:** в demo-режиме `getToken()` возвращает null (реальной сессии нет,
demo-пользователь инжектится без токена). Раньше это ломало auth-гейты:
`writeToSeller` в `ads.$id.tsx` и `isAuthed` в `ads.index.tsx` редиректили на
`/login`. Исправлено: гейты стали demo-aware (`!getToken() && !isDemoMode()` /
`getToken() || isDemoMode()`). Также: `makeSeller` (`mock.ts`) теперь включает
`numericId` (иначе «Написать» падало с «Не удалось открыть диалог»);
`demoMyListings` включает созданные в сессии объявления; кнопка
«К моим объявлениям» в success-модалке ведёт на `/my-ads`, а не в каталог.
**Важно для прод-интеграции:** когда подключат реальную auth (не demo),
`getToken()` вернёт настоящий токен, `isDemoMode()` будет false на прод-хосте,
и гейты снова заработают штатно (редирект неавторизованных на /login).

## 16. Сброс пароля — новый пароль не подходит для входа (P0-баг, требует расследования на бэке)

**Задача:** P0-баг из аудита (2026-07-08) — «после смены пароль не подходит
для входа». Расследовал максимально доступными безопасными средствами
(без email/лог-доступа к продакшену — SSH на прод для чтения `.env`/логов
был заблокирован автономность-классификатором как риск утечки секретов в
транскрипт, и я не стал обходить это ограничение).

**Что проверено и подтверждено корректным (НЕ причина бага):**

1. **Контракт запроса/ответа фронта строго совпадает с `docs/openapi/openapi.json`.**
   `ResetPasswordRequest`: `{ token, email, password, password_confirmation }`
   (все 4 поля required) — именно так и отправляет `resetPassword()` в
   `lib/api/auth.ts`. Несовпадения имён полей нет.
2. **Backend корректно валидирует токен.** Прямой запрос curl'ом с заведомо
   невалидным токеном для реального сид-аккаунта (`demo@modelizmclub.ru`,
   пароль `password123` из `ReferenceDataSeeder.php`):
   ```
   POST /api/v1/auth/reset-password
   {"email":"demo@modelizmclub.ru","token":"obviously-invalid-token-12345","password":"NewPassword999!","password_confirmation":"NewPassword999!"}

   → HTTP 422
   {"message":"passwords.token"}
   ```
   Это ожидаемый, правильный ответ (стандартный ключ перевода Laravel
   `Password::INVALID_TOKEN`). Значит backend НЕ принимает произвольный
   токен — базовая проверка токена работает.
3. **`forgot-password` не палит существование email** (корректная практика):
   ```
   POST /api/v1/auth/forgot-password {"email":"<fake>"}
   → HTTP 200 {"data":{"message":"Если аккаунт существует, ссылка для сброса пароля отправлена на email."}}
   ```
   Токен в ответе не возвращается (нет debug-режима, отдающего токен в теле
   ответа) — значит проверить полный happy-path (запрос → реальный токен из
   письма → resetPassword → login новым паролем) с фронта без доступа к
   реальному почтовому ящику **невозможно**.

**Что НЕ проверено (нужен доступ бэкенд-команды):**

Раз токен-валидация работает корректно (пункт 2) и контракт полей верный
(пункт 1), сама ошибка «новый пароль не подходит» скорее всего в
WRITE-пути после успешной валидации токена — т.е. в контроллере/сервисе,
который обрабатывает `POST /auth/reset-password` **после** прохождения
проверки токена. Прошу бэкенд-команду проверить на сервере (с доступом к
реальным письмам/логам):

1. Реально ли `Password::reset()`/аналог сохраняет новый хэш пароля в БД
   (`$user->save()` вызывается? транзакция коммитится?) — самая частая
   причина такого симптома.
2. Нет ли кэширования модели `User` (Redis/Laravel cache), из-за которого
   `/auth/login` может читать устаревший (закэшированный) хэш пароля
   сразу после успешного reset — кэш не инвалидируется при смене пароля.
3. Совпадает ли алгоритм хеширования при записи (reset) и при проверке
   (login) — например, если `Hash::make()` использует другой driver/cost
   на разных путях кода.
4. TTL токена (Laravel default — 60 минут) — не единственная причина (см.
   пункт 2, где показано, что НЕВАЛИДНЫЙ токен корректно отклоняется), но
   стоит проверить, что реальный только что запрошенный токен не считается
   протухшим сразу же (например, рассинхрон времени сервера/БД).

**Проверка на стороне бэкенда (рекомендованный сценарий):** через
`php artisan tinker` или прямой E2E с `MAIL_MAILER=log` — запросить
`forgot-password` для тестового пользователя, достать токен из
`storage/logs/laravel.log`, выполнить `reset-password` с этим токеном,
затем сразу `login` тем же паролем и убедиться, что оба шага проходят на
самом сервере (исключит любые сетевые/кэш-прокси эффекты, специфичные для
внешних вызовов).

**Frontend:** без изменений — код `lib/api/auth.ts`/`routes/reset-password.tsx`
уже соответствует документированному контракту, чинить на фронте нечего.

## 17. Публичный feature-flags endpoint (нужен для скрытия «Сообщества» через AdminSetting)

**Задача:** скрыть пункт меню «Сообщества» (Sidebar/BottomNav/лендинг) через
существующий generic-механизм `AdminSetting` (см. №21 в старом аудите —
`GET/PATCH /admin/settings`, ключ `feature.communities_enabled`).

**Проблема:** `GET /admin/settings` требует admin-авторизации — в
`docs/openapi/openapi.json` у этого пути нет оверрайда `"security": []`
(в отличие от `forgot-password`/`login`/etc, у которых он есть явно), тег
`"Admin — System"`. Значит обычный залогиненный пользователь и тем более
анонимный посетитель **не может прочитать** этот флаг — 401/403. А именно
эта аудитория (обычные посетители сайта, не админы) и должна видеть
изменённую видимость пункта меню.

Проверил также `GET /users/me/settings` — это персональные
`NotificationPreferenceResource`, не сайтовые фиче-флаги, не подходит.
Публичных `/public/*` эндпоинтов с произвольными флагами нет (есть только
`banners`, `faq`, `landing-stats`, `stats`).

**Что нужно на бэкенде:** публично читаемый endpoint (без авторизации, как
`/public/landing-stats`) для ограниченного набора white-list'нутых
site-wide флагов, например `GET /public/feature-flags` →
```json
{ "data": { "communities_enabled": false } }
```
Значение должно браться из той же таблицы `AdminSetting`
(`feature.communities_enabled`), чтобы `PATCH /admin/settings` в
админ-панели реально управлял видимостью для всех посетителей, а не только
для самого админа (который, будучи авторизован, и так может прочитать
`/admin/settings` напрямую).

**Текущее временное решение (demo/до появления backend):** тоггл
«Показывать раздел "Сообщества"» добавлен в `/admin` → Настройки →
Feature flags (демо). Хранит значение в `localStorage`
(`frontend/src/lib/config/featureFlags.ts`, ключ
`modelizm_feature_flags`), читается через хук `useFeatureFlag()` в
Sidebar/BottomNav/лендинге. **Ограничение:** значение не синхронизируется
между устройствами/браузерами — только client-side per-browser. Когда
появится `GET /public/feature-flags`, `useFeatureFlag` нужно переключить
на чтение из него (с локальным кэшем/фоллбэком на localStorage для
офлайн-режима), а сам admin-тоггл — писать через `PATCH /admin/settings`
вместо localStorage.

**Текущее frontend-решение (временное, задокументировано в коде):**
`src/lib/config/featureFlags.ts` — захардкоженная константа
`FEATURE_FLAGS.communitiesEnabled = false`. Используется в `Sidebar.tsx`,
`BottomNav.tsx`, `routes/index.tsx` (секция «Что есть в МоДелизМ») для
фильтрации пункта «Сообщества» из соответствующих массивов. Код/роуты/
компоненты самого раздела Сообщества не тронуты и не удалены — скрыта
только видимость ссылок в 3 перечисленных местах. Ссылка в верхнем меню
лендинга (`TopNav`, вне списка из задачи) намеренно не тронута — не входила
в scope этой задачи.

Когда появится публичный endpoint — заменить константу на реальный fetch
с fallback на `false` при ошибке/до ответа.

4. Отдельно: если понадобится визуально выделять сообщение с выбором

**Демо-fallback:** не нужен — MVP полностью работает в demo и real режиме
одинаково (переиспользует существующие `createConversation`/`sendMessage`,
которые уже demo-aware).

## 22. Call Seller Button — click-to-reveal номер продавца (endpoint + анти-скрапинг отложены)

**Контекст:** реализована MVP-версия «Позвонить продавцу» — номер не
присутствует в исходном HTML/JSON объявления, раскрывается только по
клику через отдельный запрос, кэшируется в памяти на сессию (не
localStorage). Ни `ListingResource`, ни `UserCompactResource`, ни
`UserResource` не содержат поля телефона нигде в текущем OpenAPI-контракте
— номер для звонка не существует в модели данных вообще, не только
скрыт.

**Нужен новый endpoint:**
`POST /listings/{id}/reveal-phone` — требует авторизации, возвращает
`{ "data": { "phone": "+7..." } }`. Это новая backend-задача (поле +
логика), а не проброс существующего поля. Формат строки `phone` —
только цифры и ведущий `+` (E.164-подобный, например `+79012345602`, без
пробелов/скобок/тире) — фронтенд строит `tel:`-ссылку напрямую из этого
значения.

**Анти-скрапинг механизм (НЕ реализован, только описание желаемого):**
- Сессионный токен на клиенте, привязанный к домену/referrer.
- Серверная валидация токена перед выдачей реального номера.
- Rate limiting на сессию/IP.
- Эвристики бот-детекции (паттерны кликов, скорость запросов, user-agent).
- Возможность отдать фейковый/decoy номер при подозрении на бота.

Это отдельный, некороткий backend-проект (генерация/валидация токенов,
инфраструктура rate limiting, бот-эвристики) и принципиально не решается
разово — постоянная гонка вооружений с скрейперами, а не одноразовая
задача. Без этого механизма MVP-эндпоинт защищён только тем, что номер
не встроен в первоначальный payload — скрипт, повторяющий
клик-и-запрос, всё ещё может его вытащить. Это осознанный, явно
зафиксированный компромисс упрощённого варианта, выбранного в этом раунде
(вместо полной подмены номера через партнёра телефонии — то была
альтернатива Б, отдельная гораздо более крупная интеграция, не в scope).

**Demo-fallback:** полностью работает — `revealSellerPhone` в
`frontend/src/lib/api/listings.ts` возвращает демо-номер из
`AdSeller.phone` (см. `SELLER_PHONES` в `frontend/src/lib/mock.ts`) с
искусственной задержкой, без реального сетевого запроса.

| `viewed_at` | `timestamp` | нет | Момент засчитанного просмотра. |

Эта таблица — рекомендация для реализации дедупликации (см. эндпоинт
`POST /videos/{id}/view`), а не обязательная к использованию фронтендом
модель — фронтенд с ней напрямую не работает, только с `views_count` на
`Video`.

### Эндпоинты

Все пути ниже — как их реально вызывает `frontend/src/lib/api/reviews.ts`
(проверено построчно против кода). Базовый префикс API (`/v1` и т.п.)
опущен для единообразия с остальными записями этого файла — использовать
тот же префикс, что и у существующих эндпоинтов (`/posts`, `/listings`).

#### `GET /videos`

- **Auth:** не требуется (публичный список), но если пользователь
  авторизован — сервер должен вернуть `is_liked` относительно него.
- **Query-параметры:**
  - `q` (string, опционально) — полнотекстовый поиск по `title`/`description`/`tags`.
  - `category` (string, опционально) — `VideoCategory.slug`. Фронтенд **никогда** не шлёт `"all"` (это чисто UI-сентинел для таба «Все» — при выборе этого таба параметр вообще не передаётся).
  - `featured` (`1` | не передаётся, опционально) — при `1` вернуть только `is_featured=true`.
  - `sort` (string) — фронтенд всегда шлёт `"new"` (сортировка по `published_at desc`). Зарезервировать как enum на будущее (`"popular"`, `"views"` и т.п.), но реализовать как минимум `"new"`.
  - `per_page` (integer) — фронтенд шлёт `50`.
  - `page` (integer, опционально) — стандартная пагинация, фронтенд сейчас не листает (грузит один раз 50 штук), но API должен быть пагинируемым для будущего.
- **Ответ `200`:**
  ```json
  {
    "data": [ /* массив ApiVideo, см. форму ниже */ ],
    "meta": { "current_page": 1, "last_page": 1, "total": 12 }
  }
  ```
  Форма одного элемента (`ApiVideo`, snake_case — именно так её парсит `mapVideo()` в `reviews.ts`):
  ```json
  {
    "uuid": "...",
    "title": "...",
    "description": "...",
    "category": { "id": "...", "slug": "..." },
    "poster_url": "https://...",
    "video_url": "https://...",
    "duration_seconds": 245,
    "views_count": 1032,
    "is_featured": false,
    "tags": ["fpv", "review"],
    "published_at": "2026-07-01T12:00:00Z",
    "uploader": { "uuid": "..." },
    "status": "published",
    "likes_count": 12,
    "comments_count": 3,
    "is_liked": false
  }
  ```
  Все поля, кроме `uuid`/`title`, опциональны на уровне контракта —
  `mapVideo()` подставляет дефолты (`""`, `0`, `false`, `[]`) при
  отсутствии, поэтому бэкенд может отдавать `null` вместо явного
  дефолта без риска сломать фронтенд, но лучше отдавать честные
  дефолтные значения.

#### `GET /videos/{id}`

- **Auth:** не требуется (как выше — `is_liked` зависит от сессии).
- **Path:** `{id}` — `uuid` видео.
- **Ответ `200`:** `{ "data": ApiVideo }` (та же форма, что элемент списка выше).
- **`404`**, если видео не найдено или не `published` (для неавторизованных/не-владельцев).

#### `GET /videos/categories`

- **Auth:** не требуется.
- **Ответ `200`:** `{ "data": [ { "id": "...", "slug": "...", "title": "...", "sort_order": 0 }, ... ] }` — отсортировано по `sort_order`.

#### `POST /videos`

- **Auth:** обязательна, роль admin или владелец-продавец (уточнить у бэкенда точную политику — минимум: авторизованный пользователь с ролью, разрешающей загрузку обзоров; UI размещения обзора сейчас доступен из профиля продавца/админки).
- **Request body** (JSON, ровно как шлёт `uploadVideo()`):
  ```json
  {
    "title": "string, required",
    "description": "string, optional",
    "category_id": "uuid, required",
    "tags": ["string", "..."],
    "poster_media_id": "uuid, required — id объекта, полученного через POST /media",
    "video_media_id": "uuid, required — id объекта, полученного через POST /media",
    "is_featured": false
  }
  ```
- **Ответ `201`:** `{ "data": ApiVideo }` — созданная запись. `status` в ответе, скорее всего, будет `"processing"`, если бэкенд делает асинхронный транскодинг/модерацию (см. «Инфраструктура» и «Модерация» ниже) — фронтенд уже умеет отображать оба статуса.
- **`422`** при невалидных `category_id`/`poster_media_id`/`video_media_id` (не найдены/не принадлежат нужному `purpose`, см. Media ниже).
- **`403`**, если у пользователя нет прав на создание обзора.

#### `PATCH /videos/{id}`

- **Auth:** admin/owner (тот же уровень, что и `POST /videos`, либо ограничить только владельцем записи + admin).
- **Request body:** частичное обновление, на практике сейчас фронтенд шлёт только `{ "is_featured": boolean }` (переключатель в админ-разделе через `setVideoFeatured()`), но эндпоинт должен поддерживать частичное обновление остальных полей (`title`/`description`/`category_id`/`tags`) для полноты CRUD, даже если UI для этого сегодня не существует.
- **Ответ `200`:** `{ "data": ApiVideo }` — обновлённая запись.
- **`403`/`404`** как обычно.

#### `DELETE /videos/{id}`

- **Auth:** admin/owner.
- **Ответ `204`**, без тела.
- **`403`/`404`** как обычно.

#### `POST /videos/{id}/view`

- **Auth:** не требуется (публичный счётчик), но если есть сессия — учитывать `user_id`.
- **Request body:** нет (пустой POST). Фронтенд вызывает это один раз при открытии детальной страницы видео (`incrementVideoView()`), без данных о позиции воспроизведения.
- **Дедуп/идемпотентность:** это счётчик «открытий страницы обзора», не аналитика точного просмотра — тем не менее нужно защититься от накрутки повторными вызовами. Рекомендация: дедуп-окно по `(video_id, user_id | session_hash)` — например, не увеличивать `views_count` повторно для того же зрителя в течение 30 минут (записывать в `VideoView`, проверять перед инкрементом). Реализация окна — на усмотрение бэкенда, но контракт эндпоинта (метод/путь/ответ) должен остаться прежним, чтобы фронтенд не менялся.
- **Ответ `200`** (или `204`) — фронтенд не читает тело ответа, только проверяет успешность запроса. Рекомендуется `200` с `{ "data": { "views_count": <новое значение> } }` для удобства будущей отладки, но это не обязательный контракт.

#### Социальные эндпоинты — точная зеркальная копия `/posts/{id}/...`

Реализуются на том же полиморфном механизме (reactable/commentable), что
уже используется у постов (см. записи файла про Feed) — просто зарегистрировать
`Video` как ещё один `reactable_type`/`commentable_type`.

- **`POST /videos/{id}/react`** — auth обязательна. Body отсутствует. Ставит лайк от текущего пользователя. Ответ `200`/`204`.
- **`DELETE /videos/{id}/react`** — auth обязательна. Снимает лайк. Ответ `200`/`204`.
  (Именно так вызывает `reactToVideo(uuid, on)`: `POST` при `on=true`, `DELETE` при `on=false` — ровно как `reactToPost`.)
- **`GET /videos/{id}/comments`** — auth не требуется. Ответ: `{ "data": [Comment, ...] }`, где `Comment` — идентичная структура объекту, который уже отдаёт `GET /posts/{id}/comments` (`id`, `authorId`/автор, `time`, `text`, `likes`, `replies`). Фронтенд не пагинирует явно, но ответ должен оставаться пагинируемым (`Paginated<Comment>` на клиенте).
- **`POST /videos/{id}/comments`** — auth обязательна. Request body: `{ "body": "string, required", "parent_uuid": "uuid, optional — для ответа на комментарий" }`. Ответ `201`: `{ "data": Comment }` — созданный комментарий, той же формы, что у `POST /posts/{id}/comments`.

#### Media (`POST /media`) — расширение существующего эндпоинта

Эндпоинт `POST /media` уже существует и документирован в предыдущих
записях этого файла для изображений. Для видео-обзоров используется тот
же эндпоинт с `purpose: "review_video"` для самого видеофайла. Постер
(обложка) загружается отдельным вызовом — **фронтенд сейчас шлёт его с
`purpose: "post"`** (переиспользует уже существующий image-purpose, т.к.
постер — обычная картинка). Бэкенд может принять `"post"` как есть, либо,
если для аккуратного учёта нужен выделенный purpose (`"review_poster"`),
завести его и сообщить — тогда фронтенд-константу `MediaPurpose` обновят.
Специфика для видео,
которую нужно явно реализовать (не наследуется автоматически из
image-flow):

- **Максимальный размер файла:** рекомендация — не более 200 МБ на
  ролик (нужно согласовать с реальной инфраструктурой хранения/CDN,
  см. ниже); отдать `413 Payload Too Large` при превышении с телом
  `{ "message": "..." }`.
- **Допустимые MIME-типы:** `video/mp4`, `video/webm`, `video/quicktime` (как минимум `video/mp4` обязателен, это основной формат, который отдаёт `<video>` в браузере без дополнительного транскодинга).
- **Серверный транскодинг/генерация превью:** на первом этапе транскодинг НЕ обязателен — фронтенд ожидает, что `video_url` — это прямая ссылка на воспроизводимый файл (браузер отдаёт `<video src="...">"`). Если сервер не транскодирует, отдавать оригинальный файл как есть и требовать от загружающего заранее сжатый/веб-совместимый MP4 (H.264/AAC). Генерация постера (`poster_media_id`) на первом этапе — на стороне клиента (canvas-снимок кадра), сервер её не обязан генерировать сам, но должен принимать отдельно загруженный image-объект как постер. Если бэкенд хочет автоматизировать (server-side thumbnail extraction, транскодинг в несколько разрешений) — это отдельный инфраструктурный проект, см. следующий раздел.

### Модерация

Раздел админки уже содержит полиморфную очередь модерации
(`ModerationType` union во фронтенде — сейчас `"posts" | "communities"`,
на бэкенде — Laravel `moderatable_type` morph), с эндпоинтами
`GET /admin/moderation` (список) и
`/admin/moderation/{type}/{id}/approve|reject|revise`.

Для видео-обзоров нужно:

1. **Зарегистрировать `Video` как `moderatable_type`** на бэкенде (тот же
   механизм, что уже применён к постам/сообществам).
2. **`GET /admin/moderation`** должен уметь возвращать элементы очереди с
   `moderatable_type: "videos"` вперемешку с существующими типами (или
   через фильтр `?type=videos` — как уже работает для `posts`/`communities`).
3. Поддержать эндпоинты `POST /admin/moderation/videos/{id}/approve`,
   `POST /admin/moderation/videos/{id}/reject`,
   `POST /admin/moderation/videos/{id}/revision` — с теми же request/response
   формами, что уже приняты для `posts`/`communities` (тело/ответ уже
   задокументированы в соответствующих более ранних записях этого файла —
   для видео повторить один в один, меняя только `{type}` в пути на
   `"videos"`).
4. **Честная оговорка про «Пожаловаться»:** кнопка жалобы на странице
   обзора сейчас — toast-заглушка («Спасибо, жалоба отправлена»), без
   реального сетевого вызова. Это **не пробел, специфичный для видео** —
   у постов сегодня та же ситуация: ни для одного типа контента в
   продукте не существует API отправки жалобы пользователем
   (`POST /reports` или аналог). Реализация полноценного цикла
   «жалоба пользователя → попадание в очередь модерации» —
   отдельная задача, которая по-хорошему должна быть реализована один раз
   универсально (полиморфно, как и остальная модерация), а не отдельно
   для видео — сделать её видео-специфичной было бы неверным
   архитектурным решением, раз тот же пробел уже существует у постов.

### Инфраструктура: хранение/транскодинг/CDN видео — отдельный проект, НЕ реализовано

Явно фиксируется как существенный, отдельный от этой фичи объём работы,
не предполагаемый выполненным в рамках Task 10 или вообще в рамках этого
раунда:

- Настоящее видео-хранилище (объектное хранилище, рассчитанное на
  файлы в десятки–сотни МБ, с потоковой раздачей, не просто blob как у
  `POST /media` для картинок).
- Транскодинг-пайплайн — приведение загруженного видео к нескольким
  разрешениям/битрейтам (адаптивное качество), нормализация кодеков.
- CDN-доставка (сейчас `POST /media` для картинок — вероятно
  прямая раздача с origin/S3-подобного хранилища; для видео при
  сколько-нибудь заметной нагрузке это не подойдёт без CDN-слоя).
- Возможный переход на HLS/DASH (адаптивный битрейт по сети,
  master-плейлист вместо одного файла) — если продукт вырастет за
  пределы MVP «залил MP4 → раздали как есть».

Существующий `POST /media` для изображений категорически не рассчитан на
эту нагрузку и не должен без изменений использоваться как окончательное
решение для видео в продакшене — MVP-контракт эндпоинтов из этой записи
специально спроектирован так, чтобы не блокировать фронтенд на
готовности этой инфраструктуры (video_url — просто ссылка, откуда бы она
ни бралась), но сама инфраструктура — отдельный проект, требующий
отдельной оценки.

### DRM — честная оговорка

Никакого настоящего DRM в этой фиче не реализовано и не предполагалось.
Важно зафиксировать это явно, чтобы не создавалось ложное ощущение
защищённости контента:

- Нативный `<video src="...">` в браузере принципиально скачиваем —
  через devtools/Network-панель, через сохранение из контекстного меню,
  через простой `curl` по прямой ссылке — независимо от того, что делает
  бэкенд.
- Атрибут `controlsList="nodownload"` на `<video>` — это чисто
  UI-подсказка браузеру скрыть пункт «Скачать» из нативных контролов,
  она не является защитой и обходится тривиально (devtools, другой
  плеер, прямой запрос по URL).
- Если в будущем понадобится осмысленная защита — это отдельное
  обсуждение про подписанные URL с коротким сроком жизни
  (signed-URL-expiry), токен-гейтед HLS-раздачу или интеграцию с
  коммерческим DRM/CDN-провайдером (Widevine/FairPlay через сторонний
  сервис). Ничего из этого не запрошено и не реализовано в текущем
  раунде — сознательно отложено, не входит в scope Task 1–10.

### Demo-fallback

Полностью работает в demo-режиме без единого реального сетевого запроса:

- `frontend/src/lib/api/reviews.ts` при `isDemoMode() === true` не
  обращается к бэкенду вообще — все функции (`fetchVideos`,
  `fetchVideoCategories`, `fetchVideo`, `incrementVideoView`,
  `uploadVideo`, `reactToVideo`, `fetchVideoComments`,
  `createVideoComment`, `deleteVideo`, `setVideoFeatured`) уходят в
  соответствующие demo-функции из `frontend/src/lib/demo-data.ts`
  (`demoVideos`, `demoVideo`, `demoVideoCategories`,
  `demoFeaturedVideos`, `demoIncrementVideoView`, `demoVideoComments`,
  `demoAddVideo`, `demoDeleteVideo`, `demoSetVideoFeatured`).
- Посевные (seed) demo-видео (`mockVideos` в `frontend/src/lib/mock.ts`)
  все указывают на один и тот же встроенный в проект сэмпл-ролик
  (Big Buck Bunny, лицензия CC BY 3.0) — используется как заглушка для
  любого количества карточек в ленте, реальные разные видеофайлы не
  нужны для демонстрации UI.
- Загрузка через `uploadVideo()` в demo-режиме создаёт запись локально
  и использует `blob:`-URL (из `uploadMedia`/`URL.createObjectURL`) для
  постера и видео — без реального аплоада на сервер, превью работает
  сразу же в том же сеансе.
- Лайки/комментарии зеркалят fidelity ленты постов: лайк не переживает
  перезагрузку страницы (не персистентный в demo-хранилище), новые
  комментарии синтезируются в рамках сессии (`createVideoComment` в
  demo-ветке возвращает объект локально, не сохраняя его на бэкенде).

## 24. Настройки (Personal Cabinet) — аккаунт, уведомления, кошелёк, реквизиты, рейтинг, история

Frontend added a `/settings/*` personal cabinet with working demo UI (localStorage / static mock). The following endpoints are needed to make each section real. All require an authenticated session (Bearer token) unless noted.

### 24.1 Смена пароля
`POST /account/change-password`
- Auth: required
- Body: `{ "current_password": string, "new_password": string }`
- Response 200: `{ "ok": true }`; 422 on wrong current password / weak new password.
- Frontend: `settings.account.tsx` submitPassword (currently toast-stub).

### 24.2 Смена email (со подтверждением)
`POST /account/change-email`
- Auth: required
- Body: `{ "new_email": string }`
- Response 202: `{ "pending": true }` — triggers a verification email; email changes only after the user confirms via the emailed link (`POST /account/confirm-email` or a tokened GET).
- Frontend: `settings.account.tsx` submitEmail (currently toast-stub).

### 24.3 Настройки уведомлений
`GET /account/notification-preferences` → `{ "friend_requests": bool, "comments": bool, "likes": bool, "messages": bool, "subscription_posts": bool }`
`PUT /account/notification-preferences` with the same shape → 200.
- Auth: required. Also drives real delivery gating (push/email) server-side.
- Frontend: `settings.notifications.tsx` + `lib/settings-prefs.ts` (currently localStorage `modelizm_notif_prefs`).

### 24.4 Кошелёк
`GET /wallet` → `{ "balance": number, "currency": "RUB" }`

- Auth: required.
- Frontend: `settings.wallet.tsx` (currently `mockWalletBalance` / `mockWalletOperations`). Top-up/withdraw actions are a later scope.

### 24.5 Реквизиты для документов сделок
`GET /account/requisites` → `{ "full_name": string, "inn": string, "phone": string, "address": string }`
`PUT /account/requisites` with the same shape → 200.

- Frontend: `settings.requisites.tsx` + `lib/settings-prefs.ts` (currently localStorage `modelizm_requisites`).

### 24.6 Рейтинг и отзывы
`GET /users/{id}/rating` → `{ "average": number, "count": number }`

- Auth: reviews are public-readable for a given profile; the owner sees their own aggregate in settings.
- Frontend: `settings.rating.tsx` (currently `mockMyRating` / `mockMyReviews`).

## Billing / оплата подписки (seller-cabinet-v1, Stage 1)

Фронт подключил **реальный** флоу оплаты подписки вместо заглушки-toast
(`routes/subscription.tsx` + новый клиент `src/lib/api/payment.ts`). Все
перечисленные эндпоинты **уже реализованы** в `backend/app/Modules/Billing`
— это не запрос «сделать эндпоинт», а фиксация контракта, который теперь
дёргает фронт, плюс два backend/infra-действия для запуска.

### Что вызывает фронт

- `POST /payments` — тело `{ "plan_slug": string, "idempotency_key": string }`
  → `201 { "data": { "payment_uuid", "checkout_url": string|null, "status", "provider" } }`.
  - `checkout_url` не null (vtb/yookassa) → фронт делает `window.location.href = checkout_url`.
  - `checkout_url` null (провайдер `stub`) → фронт вызывает `POST /payments/{uuid}/confirm-stub`.
- `GET /payments/{uuid}` — статус платежа.
- `POST /payments/{uuid}/sync` — пересинхронизировать статус у провайдера.
- `POST /payments/{uuid}/confirm-stub` — подтвердить stub-платёж (только dev/test-контур, провайдер `stub`).
- `GET /users/me/subscription` — текущая подписка или `data: null` на бесплатном тарифе.

### Возврат с хостинговой страницы провайдера

`config/billing.php` уже задаёт `return_url = FRONTEND/subscription?payment=success`
и `fail_url = .../subscription?payment=failed`. Фронт (`SubscriptionPage`)
читает `?payment=success|failed` на маунте, показывает тост и очищает
параметр. Менять эти URL не нужно — если фронтовый роут `/subscription`
сохранится, всё сойдётся.

### ⚠️ Действия на стороне backend/infra (без них флоу не заработает)

1. **Слаги тарифов должны совпадать.** Фронт шлёт `plan_slug` из
   `src/lib/config/pricing.ts` — это `"month"`, `"half"`, `"year"`. Бэкенду
   нужно засеять `subscription_plans.slug` **ровно этими значениями** (или
   согласовать иной маппинг и обновить фронтовый конфиг). Сидера
   `subscription_plans` в репозитории не нашёл — вероятно, план ещё не
   засеян.
2. **Включить эквайринг в тест-контуре.** Оба провайдера по умолчанию
   `enabled=false` → сейчас резолвится `stub`. Для боевого редиректа нужны
   env-ключи ВТБ (`VTB_ACQUIRING_*`, `VTB_ACQUIRING_ENABLED=true`) или ЮKassa
   (`YOOKASSA_*`, `YOOKASSA_ENABLED=true`).

### Не входит в Stage 1

- **Demo-режим** (neeklo/local) намеренно **не** вызывает эти эндпоинты —
  показывает честное «Оплата будет доступна после подключения эквайринга»
  (фейкового успешного платежа нет).
- **`PaymentModal.tsx`** намеренно **не** используется: реальный флоу —
  редирект на хостинговую страницу провайдера, а этот компонент —
  self-labeled прототип, который имитирует успешную оплату. Оставлен как
  dead-code (нет импортёров); удаление/замена — отдельным решением.
- **Разовое размещение (99 ₽)** — `Done`: `POST /payments` с `payable_type: "listing_placement"`,
  после оплаты начисляется `users.listing_placement_credits`.
- **Смоук-тест** «кнопка → вебхук → смена статуса подписки» помечен

## Реквизиты выплат — ручной вывод, без эквайринга (seller-cabinet-v1, Stage 2, вторая половина)

**Решение по вопросу к Никите (2026-07-12):** на старте — ручные переводы,
без ЮKassa Marketplace API и без эскроу-интеграции. Админ читает номер
карты из БД и переводит деньги вручную. Это осознанно самый простой вариант:
одно зашифрованное поле, никакой провайдерской интеграции для выплат.

Новая карточка «Карта для выплат» на `/settings/requisites` (отдельно от
существующих ФИО/ИНН/телефон/адрес из §24.5 — те остаются как есть, это
реквизиты для документов сделки, а не для перевода денег). Фронт: функции в
`src/lib/api/payout-requisites.ts`.

### Контракт (нужно реализовать на бэке)

- `GET /account/payout-requisites` → `{ "data": { "card_last4": string | null } }`.
  **Полный номер карты сюда никогда не возвращать** — только последние 4
  цифры (или `null`, если карта ещё не сохранена). Фронт использует это
  только для строки «Сейчас сохранена карта •••• 1234», не для
  предзаполнения поля ввода.
- `PUT /account/payout-requisites` с телом `{ "card_number": string }`
  (только цифры, 16 символов, фронт уже чистит от пробелов) → `200`.
  Перезаписывает сохранённый номер. Валидация формата (Luhn/длина) —
  на бэке.

### 🔒 Безопасность (обязательное требование)

- Хранить номер карты **в одном зашифрованном поле** (например, Laravel
  `encrypted` cast на колонке `payout_card_number`, ключ — `APP_KEY`/отдельный
  KMS-секрет, не в общей БД-дампе в открытом виде). Это НЕ то же самое, что
  токенизация у эквайринга (см. «Способы оплаты» выше) — здесь мы
  сознательно берём on-file PAN, потому что перевод делает человек, а не
  API. Оценить, требует ли это PCI DSS SAQ-уровня для проекта — юридический
  вопрос вне фронтового скоупа, но enable field-level encryption обязательно
  до продакшна.
- Полный номер карты никогда не должен попадать в логи запросов/ошибок.
- Доступ к расшифрованному значению — только у админ-панели (для ручного
  перевода), не у обычных API-ответов пользователю.

### Демо-режим

`isDemoMode()` → номер карты не отправляется на сервер, кладётся в
`localStorage` (`modelizm_payout_requisites`), GET-эквивалент читает оттуда
и маскирует до last4 на лету. На проде это поведение не задействуется —
реальный режим всегда идёт через сервер.

### Смоук-тест — pending

Сохранить карту → перезагрузить страницу → убедиться, что показывается
верный last4, а не полный номер. Нужен работающий backend-эндпоинт.

### Гидратация: React error #418 (SSR/CSR mismatch) — не блокер

- **Симптом:** в консоли браузера на многих страницах (лендинг, `/feed`,

- **От бэка ничего не требуется**, если только корневой причиной не окажется
  поле, которое сервер отдаёт в непредсказуемом порядке/формате (тогда заведём

## 26. Иконки — загрузка кастомных SVG, санитизация, централизованная перекраска (админка, фаза 2b)

**Спека:** `docs/superpowers/specs/2026-07-14-admin-icons-svg-recolor-design.md`.

Админ загружает собственные SVG-иконки, назначает их на именованные UI-слоты и
централизованно перекрашивает через 9 токенов UI Kit 2.0, с публикацией «для
всех» + предпросмотром + откатом. Модель `icon_overrides` переиспользует
`SystemSetting` + `PUT /admin/settings` + audit log (как Design System
colors/fonts), новых write-эндпоинтов для карты назначений не нужно.

### Что нужно на бэкенде

| # | Что | Тип | Детали |
|---|-----|-----|--------|
| 1 | Модель+миграция `IconAsset` | новое | `id, name (string), svg (text — санитизированная монохромная разметка), source (enum: upload), uploaded_by (fk users), created_at, deleted_at (nullable — мягкое удаление)` |
| 2 | **Санитизация + токенизация SVG при загрузке** | новое, **критично (XSS)** | whitelist тегов (`svg,g,path,circle,ellipse,rect,line,polyline,polygon,defs,title,desc`) и атрибутов; вырезать `<script>`, любые `on*`, `<foreignObject>`, `<style>`, внешние `href/xlink:href`, `<use>`/`<image>` на внешние ресурсы, `javascript:`/`data:` URI; убрать конкретные `fill/stroke` → `currentColor`; отклонить многоцветные (после нормализации остаётся >1 визуального цвета / есть градиенты) с валидационной ошибкой |
| 3 | `POST /media` `purpose = icon` | расширение существующего | принять `icon`, прогнать файл через санитайзер из п.2, создать `IconAsset` и **вернуть его целиком в теле ответа** — `{ "data": { "id", "name", "svg" (санитизированная+токенизированная разметка), ... } }`. Фронт читает `IconAsset` прямо из этого ответа и **не делает никакого follow-up `GET`** (отдельного `GET /admin/icon-assets/{id}` не существует и не нужен — см. п.4/5) |
| 4 | `GET /admin/icon-assets` | новое, admin-only | список библиотеки иконок (не удалённых) для админки. Это единственный `GET`-эндпоинт для чтения `IconAsset` — постранично/списком, **без** отдельного `GET /admin/icon-assets/{id}` для одного элемента |
| 5 | `DELETE /admin/icon-assets/{id}` | новое, admin-only | мягкое удаление (проставить `deleted_at`); в фазе 3 станет корзиной медиаменеджера |
| 6 | `GET /icon-overrides` | новое, **публичное (без авторизации)** | по образцу `GET /feature-flags` / планируемого `GET /design-system`; отдаёт карту `{ "<slotKey>": { "assetId": "...", "svg": "<sanitized>", "token": "<tokenKey>" }, ... }` (SVG инлайном, чтобы фронт не делал N запросов; `assetId` сохраняется для откатов/повторной публикации). Если `SystemSetting` `icon_overrides` не создан — отдать `{}` (не 404, не ошибка) |

### Что уже есть и переиспользуется без изменений

- `SystemSetting` (`key='icon_overrides'`, `group='design'`,
  `value = { slotKey: { assetId, svg, token } }` — SVG хранится инлайном в
  самой настройке, а не только `assetId`/`token`, чтобы `GET /icon-overrides`
  отдавал готовую разметку одним запросом), `PUT /admin/settings` для
  публикации, `AuditService::log()` для истории, `GET /admin/audit-logs` для
  отката (читать последнюю запись с `keys` содержащим `icon_overrides`,
  публиковать `old_values`).

### Токены цвета (строго 9, произвольный hex запрещён)

`accent | foreground | success | warning | info | danger | commercial |
neutral | foreground-70` (источник — `frontend/src/styles.css` `:root`).

### Замечания по безопасности

- Санитизация обязательно **на сервере** — клиентская проверка только UX-подсказка,

- Фронт инлайнит только уже-санитизированную сервером разметку и проверяет, что

### До появления эндпоинтов

Фронт работает целиком на дефолтных lucide-иконках (fallback гарантирован):
