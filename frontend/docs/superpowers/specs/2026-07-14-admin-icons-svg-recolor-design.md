# Админка: Иконки — загрузка SVG, реестр слотов, централизованная перекраска токенами (фаза 2b) — Design

## Контекст

Одна из ~8 независимых подсистем в общем запросе «расширить админку до полного
управления сайтом». Порядок фаз:

- **Фаза 1 — RBAC + Audit Log** — реализована и задеплоена
  (`2026-07-12-admin-rbac-audit-log-design.md`).
- **Фаза 2 — Design System: цвета + шрифты** — спека готова, реализация ещё нет
  (`2026-07-13-admin-design-system-design.md`).
- **Фаза 2b — Иконки (эта спека)** — загрузка кастомных SVG, назначение на
  именованные UI-слоты, централизованная перекраска через 9 токенов UI Kit 2.0.
- Фазы 3–8 (медиаменеджер, feature-флаги, контент-сущности, пользователи,
  модерация, аналитика) — отдельными спеками позже.

Эта спека покрывает **только иконки**. Каждая подсистема — своя спека → план →
реализация.

### Ключевые находки при исследовании кода

1. **В UI нет PNG-иконок.** Все иконки — Lucide SVG, захардкоженные импортом
   `from "lucide-react"` в **174 файлах**. Иконки категорий хранятся как **имя
   lucide-иконки** (строка, напр. `"Boxes"`) в поле `category.icon` и рендерятся
   через `resolveLucideIcon(name)` (`frontend/src/lib/lucide-icon.ts`) и
   `CategoryIcon` (`frontend/src/components/layout/RightCategories.tsx`,
   `FeedRightRail.tsx`). PNG в проекте — только логотип и демо-картинки лендинга.
   Поэтому «перевести PNG→SVG» в текущем коде нечего: задача 2.1 — это **новая
   подсистема** загрузки кастомных SVG-иконок, а не конвертация существующих.

2. **Инфраструктура для переиспользования (без нового кода):**
   - `App\Models\SystemSetting` (`key`, `value: json`, `group`) — уже есть,
     `AdminSettingsController` пишет произвольные ключ-значение через
     `PUT /admin/settings` и логирует каждое изменение в audit log через
     `Modules\Admin\Services\AuditService::log()` (история изменений иконок
     появится автоматически, как у Design System).
   - `Modules\PublicContent\...\FeatureFlagsController` — прецедент публичного
     (без авторизации) read-only admin-controlled эндпоинта, который читают все
     посетители. `GET /icon-overrides` строится по тому же образцу, что и
     запланированный `GET /design-system`.
   - `POST /media` (`uploadMedia(file, purpose)` в
     `frontend/src/lib/api/media.ts`) — существующий рабочий аплоад; `purpose`
     расширяется значением `icon`.

3. **9 токенов цвета (UI Kit 2.0)** в `frontend/src/styles.css` `:root`:
   `accent`, `foreground`, `success`, `warning`, `info`, `danger`,
   `commercial` (`accent-commercial`), `neutral`, `foreground-70`. Перекраска
   иконок опирается строго на них — произвольный hex запрещён.

## Требования (из запроса пользователя)

- Админ загружает иконки, выбирает место (блок/страница), где иконка
  используется, и заменяет её: **загрузил → выбрал место → применил**.
- Иконки перекрашиваются через **RGB/токен-систему на бэкенде** так, что при
  смене цвета в админке он меняется **у всех пользователей**.
- Управление цветом иконок опирается на **9 токенов UI Kit 2.0**, а не на хаос
  произвольных цветов.
- Только selects/inputs/кнопки/загрузка файла — **никакого JSON**.
- Изменение применяется глобально; нужен **предпросмотр** и **откат последнего
  изменения**.

## Решения, зафиксированные при брейншторме

1. **Охват = реестр слотов (override), не полная замена.** Конечный список
   именованных переопределяемых мест; кастомный SVG переопределяет слот, иначе —
   fallback на текущую lucide-иконку. 174 файла прямых импортов не трогаем.
2. **Хранилище = мини-библиотека сейчас**, самодостаточная, на базе `POST /media`.
   Иконки не ждут медиаменеджера (фаза 3); позже переедут туда без ломки API.
3. **Цвет = один из 9 токенов**, не произвольный hex.
4. **Загружаемые иконки монохромные** (line-иконки, как lucide) — единственный
   способ, которым «единый источник цвета» работает чисто. Многоцветные
   отклоняются на загрузке.

## Архитектура

### A. Данные и хранилище

**Новая сущность `IconAsset`** (мини-библиотека загруженных иконок):

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | uuid/pk | идентификатор |
| `name` | string | человекочитаемое имя (из имени файла, редактируемо) |
| `svg` | text | санитизированная монохромная разметка (`currentColor`) для инлайн-рендера; line-иконки ~1–3 КБ |
| `source` | enum | `upload` (задел на будущие источники) |
| `uploaded_by` | fk users | кто загрузил (для audit/атрибуции) |
| `created_at` | timestamp | |
| `deleted_at` | timestamp nullable | мягкое удаление; в фазе 3 — корзина медиаменеджера |

Хранение разметки строкой (а не только файлом) нужно, чтобы публичный эндпоинт
отдавал SVG **инлайном** и фронт не делал N запросов за иконками. Оригинальный
файл при этом всё равно проходит через `POST /media`, поэтому в фазе 3 запись
`IconAsset` естественно привязывается к медиа-файлу.

**Назначения на слоты — без новой таблицы.** Один `SystemSetting`:

```
key   = 'icon_overrides'
group = 'design'
value = { "<slotKey>": { "assetId": "<IconAsset.id>", "token": "<tokenKey>" }, ... }
```

Публикация — существующий `PUT /admin/settings`
(`{ "settings": [{ "key": "icon_overrides", "value": {...}, "group": "design" }] }`),
история — автоматически в audit log. Новых write-эндпоинтов для карты не нужно.

### B. Перекраска + безопасность SVG (ядро требования)

**Токенизация (централизованный цвет).** При загрузке SVG сервер:
1. Санитизирует (см. ниже).
2. Удаляет конкретные `fill`/`stroke` и проставляет `fill="currentColor"`
   (для line-иконок со `stroke` — `stroke="currentColor"`).
3. Если после удаления цветов в SVG остаётся более одного визуального цвета
   (например, встроенные `<linearGradient>`, несколько разных `fill`, которые
   не сводятся к одному `currentColor`) — иконка считается **многоцветной** и
   **отклоняется** с понятной ошибкой валидации.

На фронте иконка рендерится внутри контейнера с
`style={{ color: "var(--<token>)" }}`. Поскольку разметка использует
`currentColor`, `fill`/`stroke` наследуют этот `color`. Смена токена слота в
админке → публикация `icon_overrides` → `GET /icon-overrides` отдаёт новый
`token` → цвет меняется у всех, без деплоя. Токен строго из 9:
`accent | foreground | success | warning | info | danger | commercial | neutral | foreground-70`.

**Безопасность (критично, бэкенд).** SVG — вектор XSS (может содержать скрипты).
Обязательная **серверная** санитизация по белому списку при загрузке:
- разрешённые теги: `svg, g, path, circle, ellipse, rect, line, polyline,
  polygon, defs, title, desc` (list уточняется на реализации, но строго
  whitelist, не blacklist);
- вырезать `<script>`, любые `on*`-атрибуты, `<foreignObject>`, `<style>` с
  выражениями, внешние `href`/`xlink:href` (оставить только безопасные
  внутренние), `<use>` на внешние ресурсы, `<image>` с внешним src;
- запретить `javascript:`/`data:`-URI в атрибутах.

Санитизация — на **бэкенде** (клиентская проверка — только UX-подсказка, не
граница безопасности). Инлайн-рендер санитизированной разметки на фронте — через
контролируемую вставку (напр. `dangerouslySetInnerHTML` **только** для разметки,
пришедшей с уже-санитизировавшего сервера); фронт дополнительно проверяет, что
корневой узел — `<svg>`, и не рендерит ничего иного.

### C. Реестр слотов + рендер на фронте

**`ICON_SLOTS`** — типизированный конечный реестр в коде
(`frontend/src/lib/icon-slots.ts`, новый файл):

```ts
export interface IconSlot {
  key: string;                 // стабильный ключ, напр. "nav.feed"
  label: string;               // человекочитаемо: «Навигация — Лента»
  group: "nav" | "section" | "category";
  defaultLucide: string;       // имя lucide-иконки по умолчанию
  defaultToken: TokenKey;      // токен цвета по умолчанию
}
export const ICON_SLOTS: IconSlot[] = [ /* явный конечный список */ ];
```

- **Статические слоты** (навигация, заголовки разделов) — перечислены явно.
  Первый набор для фазы 2b: пункты сайдбара/бот-нава и заголовки крупных
  разделов (точный список фиксируется в implementation plan — берётся из
  существующих `nav.tsx`/`Sidebar.tsx`/`FeedRightRail`).
- **Иконки категорий** — динамические (по строке в БД). Расширяем формат
  `category.icon`: значение либо имя lucide (как сейчас), либо `asset:<IconAsset.id>`.
  `resolveLucideIcon`/`CategoryIcon` учат распознавать префикс `asset:` и
  рендерить загруженную иконку. **Механизм** — в этой фазе; сам пикер иконки
  категории (выбор в форме категории) может остаться в фазе 5 (управление
  категориями) — в фазе 2b категории переопределяются через общий слот-UI
  (группа «Категории», слоты `category:<id>`).

**Компонент `<Icon slot="..." fallbackClassName=... />`**
(`frontend/src/components/ui/Icon.tsx`, новый):
- читает опубликованную карту `icon_overrides` (см. бутстрап ниже);
- если для `slot` есть override → инлайнит `svg` из override с цветом
  `var(--<token>)`;
- иначе → рендерит `defaultLucide` слота с `defaultToken`.
- **Fallback гарантирован** — пустых мест не бывает, даже если сервер недоступен
  или override не задан.

**Публичное чтение.** Новый `GET /icon-overrides` (без авторизации, по образцу
`GET /design-system`) → `{ "<slotKey>": { "svg": "<sanitized>", "token": "accent" }, ... }`
(SVG инлайном). Если запись `SystemSetting` `icon_overrides` не создана —
отдаёт `{}` (не 404, не ошибка): фронт полностью работает на lucide-дефолтах.

**Бутстрап.** В не-demo режиме на старте фронт фетчит `GET /icon-overrides` и
кладёт карту в модуль-стор (по аналогии с планируемым фетчем `/design-system` в
`theme-manager.ts`). В demo-режиме (`isDemoMode()`) серверный фетч пропускается —
demo-стенд остаётся на дефолтах/локальном черновике. `<Icon>` подписан на стор
и перерисовывается при публикации.

### D. Админ-UX (no-JSON)

Вкладка **«Иконки»** внутри существующего `DesignSystemSection`
(`admin.tsx`) — цвета/шрифты/иконки в одном разделе Design System. Три зоны:

1. **Библиотека** — сетка `IconAsset` + кнопка «Загрузить SVG».
   Загрузка: клиентская проверка (`.svg`, размер), `POST /media`
   (`purpose: "icon"`) → сервер санитизирует+токенизирует+создаёт `IconAsset` →
   иконка появляется в сетке (уже монохромная, в `currentColor`). Многоцветный
   SVG → ошибка «иконка должна быть одноцветной». Удаление иконки —
   `DELETE /admin/icon-assets/{id}` (мягкое).

2. **Назначение на слот** (рабочий процесс «загрузил → выбрал место → применил»):
   - `<select>` **Слот** (группы: Навигация / Разделы / Категории), показывает
     `label`, не `key`.
   - `<select>` **Иконка** — из библиотеки или «по умолчанию (lucide)» (сброс
     слота).
   - `<select>` **Цвет** — один из 9 токенов, с превью-кружком.
   - «Применить» → запись в **черновик** карты `icon_overrides` (локально).

3. **Превью + публикация** (тот же паттерн, что colors/fonts):
   - **Живое превью** — черновик применяется только в браузере админа (локальный
     оверлей поверх опубликованной карты). Другие пользователи не видят.
   - **«Опубликовать для всех»** → `PUT /admin/settings` (`icon_overrides`) →
     `GET /icon-overrides` отдаёт новую карту → меняется у всех. Пишется в audit
     log автоматически.
   - **«Откатить последнее изменение»** — читает последнюю audit-запись по
     `icon_overrides` (`GET /admin/audit-logs`, `action = 'admin.settings.update'`,
     `keys` содержит `icon_overrides`), публикует `old_values`. Кнопка **честно
     скрыта**, если откатывать нечего. То же ограничение, что в спеке colors/fonts:
     точный формат `old_values` нужно проверить по факту (реальный ответ
     `AuditService::log()`) — если предыдущее значение `icon_overrides` нельзя
     однозначно вычленить (батч-апдейт с другими ключами), кнопка «Откатить»
     скрывается, а не откатывает частично. Pending-заметка для плана.

**Права (фаза 1 RBAC).** Вкладка Design System уже `roles: ["admin"]` — модератор
её не видит. Перекраска/замена иконок для всех — только супер-админ.

**Мобильно.** Сетка библиотеки — адаптивный `grid`; зона назначения —
вертикальный стек селектов; проверка overflow 360/390/430. Админка desktop-first,
но раздел не должен ломаться на узком экране.

## Сводка: бэкенд vs фронтенд

**Бэкенд (документируется в `backend-endpoints-needed.md`, реализует бэкендер):**

| Что | Тип | Примечание |
|-----|-----|-----------|
| `IconAsset` — модель + миграция | новое | поля из раздела A |
| Санитизация + токенизация SVG на загрузке | новое, **критично** | whitelist тегов/атрибутов; вырезать script/on*/внешние href/foreignObject; `fill/stroke → currentColor`; отклонять многоцветные |
| `POST /media` `purpose = icon` | расширение | прогон через санитайзер, создание `IconAsset` |
| `GET /admin/icon-assets` | новое | список библиотеки (admin-only) |
| `DELETE /admin/icon-assets/{id}` | новое | мягкое удаление (admin-only) |
| `GET /icon-overrides` | новое, **публичное** | по образцу `GET /design-system`; карта `slot → { svg, token }` |
| `SystemSetting icon_overrides`, `PUT /admin/settings`, audit log | существующее | без изменений |

**Фронтенд (реализуется после одобрения плана):**
- `frontend/src/lib/icon-slots.ts` — реестр `ICON_SLOTS`.
- `frontend/src/components/ui/Icon.tsx` — `<Icon slot=... />` с override→fallback.
- `frontend/src/lib/api/icons.ts` — обёртки `fetchIconOverrides`, `fetchIconAssets`,
  `deleteIconAsset`, `publishIconOverrides` (через существующий settings-эндпоинт).
- Бутстрап `icon_overrides` в не-demo (стор + подписка), по аналогии с design-system.
- Расширение `resolveLucideIcon`/`CategoryIcon` для `asset:<id>`.
- Вкладка «Иконки» в `DesignSystemSection` (библиотека + назначение + превью +
  публикация + откат).

## Тестирование / приёмка

- `tsc --noEmit` чист.
- Live-проверка на demo-стенде:
  - Админ загружает валидный монохромный SVG → появляется в библиотеке в
    `currentColor`.
  - Загрузка многоцветного SVG → отклоняется с понятной ошибкой.
  - Назначение иконки на слот + токен → **живое превью** только у админа (другая
    вкладка/инкогнито под другим пользователем не видит до публикации).
  - «Опубликовать для всех» → `GET /icon-overrides` отдаёт новую карту → у нового
    посетителя (свежая загрузка, не demo) слот показывает загруженную иконку в
    цвете токена.
  - Смена токена слота + публикация → цвет иконки меняется у всех.
  - Новая запись в audit log (раздел «История изменений», фаза 1).
  - «Откатить последнее» возвращает предыдущую карту, создаёт свою audit-запись;
    если публикаций не было — кнопка скрыта.
  - Fallback: слот без override рендерит lucide-дефолт; при недоступном
    `GET /icon-overrides` весь UI остаётся на дефолтах, ничего не пропадает.
  - Overflow-пробы 360/390/430 на вкладке «Иконки».
- Безопасность: загрузка SVG со встроенным `<script>`/`onload` → сервер вырезает,
  во фронте не исполняется (проверить, что сохранённая разметка чистая).

## Не входит в эту фазу

- Полный медиаменеджер с папками/иерархией/корзиной (фаза 3) — иконки пока
  плоская библиотека с мягким удалением.
- Полная замена всех 174 lucide-импортов — не требуется, только именованные слоты.
- Произвольный hex-цвет иконок — запрещён намеренно, только 9 токенов.
- Пикер иконки в форме редактирования категории — механизм `asset:<id>` готов в
  этой фазе, но пикер внутри CRUD категорий может переехать в фазу 5.
- Многоцветные/растровые иконки — вне модели централизованной перекраски.
