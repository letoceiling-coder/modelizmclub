# Личный кабинет `/settings/*` под проверку ЮKassa — Design Spec

## Context

ЮKassa на модерации требует доступ в личный кабинет покупателя для ручной проверки менеджером. Проверка идёт на **демо-стенде (neeklo), где нет рабочего бэкенда** (весь продукт крутится на mock-данных, `isDemoMode()`). Значит кабинет должен читаться как **цельный рабочий раздел**, а не набор надписей «скоро». Честность — жёсткое требование: **никакого фейка безопасности** (2FA, «защита включена») и **никакой фейковой привязки банковских карт** (карты — на стороне ЮKassa).

Текущий аудит `/settings/*` ([SettingsNav.tsx](../../../src/components/settings/SettingsNav.tsx), 6 разделов):

| Раздел | Состояние сейчас |
|---|---|
| Профиль (публичный) | ✅ работает (ссылка на `/profile`, реальное редактирование demo) |
| Смена пароля | ⚠️ заглушка — submit → `toast("будет доступно позже")` |
| Email + повторное письмо | ⚠️ заглушка — «статус подтверждения в разработке» |
| Смена email | ⚠️ заглушка — submit → `toast("будет доступно позже")` |
| Телефон | 🟡 demo-localStorage (реально персистит) |
| Соцсети | 🟡 demo-localStorage |
| Уведомления | 🟡 demo-localStorage |
| Кошелёк | ⚠️ mock read-only («Демо-баланс») |
| Реквизиты | 🟡 demo-localStorage |
| История просмотров | ✅ работает (localStorage) |
| Рейтинг | ⚠️ mock read-only |
| Безопасность | ❌ раздела нет |

## Принцип (decided)

- Всё, что **можно честно сделать рабочим на фронте** (навигация, формы, персистентность в localStorage, валидация, понятные состояния) — работает без надписей «заглушка».
- Всё, что **требует бэкенд** (смена пароля, отправка письма подтверждения) — оформляется как **полноценная форма с валидацией + честное состояние отправки** («Запрос отправлен на ваш email»), вызывающая документированный API-endpoint, который в demo — no-op. Это правда: в бою смена пароля и идёт через подтверждение по email. Фейкового «пароль изменён» нет.
- Никакого фейка: 2FA, активные сессии, привязка карт — **не рисуем**.

## Изменения по разделам

### 1. Смена email (заглушка → честно работает на демо)
`settings.account.tsx`. `submitEmail` перестаёт быть заглушкой: обновляет email в demo-хранилище (localStorage через `setAccountExtra`/аналог) **и помечает email как неподтверждённый**, тост «Email обновлён — подтвердите по ссылке из письма». Это честно: на демо email реально меняется локально, и логично уходит в статус «не подтверждён».

### 2. Email + верификация (заглушка → честное состояние)
Блок Email показывает текущий адрес и **честный статус** («Подтверждён» / «Не подтверждён» из данных пользователя). Кнопка «Отправить письмо повторно» → честное состояние **«Письмо со ссылкой подтверждения отправлено на {email}»** (вызов документированного `POST /account/email/verify/resend`, no-op в demo). Убрать формулировку «в разработке».

### 3. Смена пароля (заглушка → форма + честное состояние отправки)
Переносится в новый раздел **Безопасность** (см. ниже). Форма (текущий/новый/повтор) с валидацией остаётся. `submitPassword` вместо `toast("будет доступно позже")` → честное состояние **«Запрос на смену пароля отправлен на {email}. Перейдите по ссылке из письма, чтобы подтвердить.»** Вызов документированного `POST /account/password/change-request` (no-op в demo). **Никакого фейкового «пароль изменён».**

### 4. Новый раздел «Безопасность» (`/settings/security`)
Добавляется в `SettingsNav`. Содержит **только реальные** вещи:
- **Смена пароля** — форма из п.3 (честное состояние отправки).
- **Заблокированные пользователи** — управление списком, который в приложении **уже реальный** (`blockedUserIds` в store; сейчас доступен из профиля). Здесь — ссылка/секция управления.
- **Нет** фейковой 2FA и «активных сессий». Если позже появится бэкенд — добавим; сейчас раздел честно ограничен реальным функционалом.

### 5. Кошелёк / Рейтинг (mock read-only → честная подпись)
Оставляем как есть по данным (demo/mock), но подписи делаем недвусмысленными, чтобы читалось как **демонстрационные данные**, а не сломанный раздел. Кошелёк уже подписан «Демо-баланс» — оставить. Рейтинг — добавить короткую сноску «Демонстрационные отзывы» при пустом реальном источнике. Никакой привязки карт в Кошельке.

### 6. Работают — не трогаем (кроме косметики подписей)
Профиль, Телефон, Соцсети, Уведомления, Реквизиты, История просмотров. Это рабочая доказательная база кабинета: формы, персист, валидация, понятные состояния — уже без «заглушек».

## Что вообще НЕ фронтенд (обозначить, не «рисовать»)

Для полноты — это не входит в спек и не имитируется в UI:
- Договор-оферта, реквизиты юрлица как юр. документы.
- Чеки по 54-ФЗ, фискализация.
- Реальная обработка платежей и привязка карт (hosted-форма ЮKassa на их стороне).
- 2FA, серверные «активные сессии».

Фронт лишь **вызывает документированные endpoints** там, где интеграция его касается (смена пароля, повторная верификация email) — код бэкенда не пишем, добавляем записи в [backend-endpoints-needed.md](../../backend-endpoints-needed.md).

## Mobile behavior (primary)

`/settings` — это rail (список разделов) + контент (`Outlet`). На мобильном список разделов заполняет экран, выбор ведёт на under-route. Новый раздел «Безопасность» встаёт в тот же список. Все формы — существующий `SettingsSectionShell`, поля/кнопки ≥44px, без горизонтального overflow (проверка probes пустая на 360/390/430). Честные состояния («Запрос отправлен…») — inline-блок под формой (не исчезающий тост), читаемый на узком экране.

## Desktop behavior

Тот же rail слева + контент справа (как сейчас, `hidden lg:*` уже есть). Новый раздел — ещё одна строка в rail. Никакой отдельной desktop-вёрстки не требуется.

## API-first (документировать, не кодить)

Добавить в `backend-endpoints-needed.md`:
- `POST /account/password/change-request` → отправляет письмо со ссылкой смены пароля; фронт показывает «Запрос отправлен».
- `POST /account/email/verify/resend` → повторная отправка письма подтверждения.
- `POST /account/email` (смена email с последующей верификацией) — на бою; в demo email меняется локально.

## Non-goals

- Фейковая 2FA, активные сессии, привязка карт — не делаем.
- Реальная смена пароля/отправка почты на демо — не фейкуем «успех»; только честное состояние отправки.
- Бэкенд-код — не пишем (API-first, только документируем).

## Testing

`npx tsc --noEmit` clean + live 360/390/430 на каждом `/settings/*` разделе:
- Оба overflow-probes пусты; страница не скроллится по горизонтали.
- Смена email: меняет отображаемый email в demo, статус → «не подтверждён».
- Смена пароля: валидация работает; submit → inline «Запрос отправлен на {email}»; фейкового «изменён» нет.
- Email: честный статус + «письмо отправлено» состояние; нет текста «в разработке».
- Новый раздел «Безопасность» доступен из rail, содержит смену пароля + заблокированных.
- Разделы, что работали (телефон/соцсети/уведомления/реквизиты/история), не сломаны.
- Нигде нет надписей «будет доступно позже».
