# Админка: Design System — цвета + шрифты (фаза 2 расширения админ-панели) — Design

## Контекст

Вторая из ~7 независимых подсистем в общем запросе «расширить админку до
полного управления сайтом» (фаза 1 — RBAC + Audit Log — реализована и
задеплоена: см. `2026-07-12-admin-rbac-audit-log-design.md`). Эта спека
покрывает **только цвета и шрифты**. Полная замена набора иконок (lucide →
другой набор) намеренно вынесена в отдельную фазу 2b — она требует
рефакторинга сотен мест прямого импорта иконок по всему коду и по объёму
сопоставима с целой отдельной фичей, не с настройкой.

### Ключевая находка при исследовании кода

Текущая вкладка «Design System» в админке (`DesignSystemSection` в
`frontend/src/routes/admin.tsx`) красит тему **только в localStorage самого
администратора** — в коде `frontend/src/lib/theme-manager.ts` прямым текстом
написано: «Persists to localStorage... does not touch app logic», и там же
уже стоит `TODO(backend): when a global "design settings" endpoint exists,
POST the chosen preset id here so every user gets the brand-selected
accent.» Это ровно то, что нужно построить.

Инфраструктура, которую можно переиспользовать без нового кода:
- `App\Models\SystemSetting` (`key`, `value: json`, `group`) — уже есть,
  уже используется `AdminSettingsController` для произвольных
  ключ-значение настроек.
- `AdminSettingsController::update()` уже вызывает
  `Modules\Admin\Services\AuditService::log()` на каждое изменение —
  значит история изменений design system **автоматически** появляется в
  Audit Log из фазы 1, без единой новой строки логирования.
- `Modules\PublicContent\Http\Controllers\Api\V1\FeatureFlagsController` —
  уже существующий прецедент публичного (без авторизации), read-only
  admin-controlled эндпоинта, который читают все посетители сайта. Design
  System получит эндпоинт по тому же паттерну.
- `frontend/src/lib/theme-manager.ts` (`applyTheme`, `applyAccentPreset`,
  `applyMode`, `loadTheme`, `saveTheme`) — вся клиентская логика применения
  темы уже написана и используется на превью-пути без изменений.

## Архитектура

### 1. Хранение

Один `SystemSetting` с `key = 'design_system'`, `group = 'design'`,
`value = { "accent": "blue" | "menthol", "font": "default" | "alt" }`.
Публикация — существующий `PUT /admin/settings` (тело:
`{ "settings": [{ "key": "design_system", "value": {...}, "group": "design" }] }`),
новых backend-эндпоинтов для записи не требуется.

### 2. Публичное чтение

Новый эндпоинт `GET /design-system` (без авторизации, по образцу
`GET /feature-flags`) → `{ "accent": "blue" | "menthol", "font": "default" | "alt" }`.
Если запись `SystemSetting` с ключом `design_system` ещё не создана —
отдаёт дефолт `{ "accent": "blue", "font": "default" }` (текущее поведение
до этой фичи), не 404 и не ошибку.

`frontend/src/lib/theme-manager.ts` `bootstrapTheme()`: в не-demo режиме
дополнительно фетчит `GET /design-system` и применяет результат поверх
локального `loadTheme()` — то есть публикация с сервера становится
источником правды для всех обычных посетителей. В demo-режиме (`isDemoMode()`)
серверный фетч пропускается — demo-стенд не имеет смысла синхронизировать
с продовой публикацией, локальный выбор остаётся как есть (текущее
поведение, не меняется).

### 3. Откат последнего изменения

Не отдельная таблица версий. При клике «Откатить последнее изменение»
фронт:
1. Читает последнюю запись из уже существующего audit log
   (`GET /admin/audit-logs`, см. фазу 1) с `action = 'admin.settings.update'`
   и `auditable_type`, соответствующим `design_system`-записи (по
   `new_values.keys` содержащим `"design_system"` — `AdminSettingsController`
   логирует `['keys' => ...]`, см. код выше).
2. Если такой записи нет — кнопка «Откатить» не отображается вовсе
   (не задизейблена молча — честно отсутствует, если откатывать нечего).
3. Если есть — публикует `old_values` того ключа как новое значение через
   тот же `PUT /admin/settings`. Это создаёт **новую** запись в audit log
   (не удаляет и не переписывает старую) — откат полностью прослеживаем,
   как обычное изменение.

Ограничение audit log сегодня: `old_values`/`new_values` хранятся per-ключ
внутри одного audit-события с `keys: [...]` (см. `AdminSettingsController::update`),
а не per-конкретное-значение с индивидуальной привязкой к `design_system`.
Если на момент реализации выяснится, что структура `old_values` в
конкретной audit-записи не даёт однозначно вычленить именно предыдущее
значение `design_system` (например, если апдейт логировался батчем с
другими ключами настроек одновременно) — реализация должна честно скрыть
кнопку «Откатить» в этом случае, а не откатывать частично или неверно.
Это pending-заметка для implementation plan: точный формат `old_values`
нужно проверить по факту (прочитать реальный ответ `AuditService::log()`
для `admin.settings.update`), не гадать заранее.

### 4. Превью

`DesignSystemSection` в `admin.tsx` переделывается: выбор пресета
(`<select>` цвет, `<select>` шрифт) вызывает `applyTheme()`/
`applyAccentPreset()` **только локально**, в браузере самого админа —
переиспользует существующую функцию без изменений, ничего не публикует.
Остальные пользователи не видят изменение, пока не нажата отдельная кнопка
«Опубликовать для всех».

### 5. Шрифты — фиксированный набор пар, не свободный ввод

Сейчас три семейства (`Manrope`/`Space Grotesk`/`IBM Plex Mono`)
захардкожены и предзагружены одним `<link>` на Google Fonts в
`frontend/src/routes/__root.tsx`. Свободный ввод шрифта админом создаёт
риск (произвольный внешний URL вне предзагруженного `<link>`, FOUC,
недоступный/нечитаемый шрифт для всех сразу). Вместо этого — фиксированный
набор из 2 пар:

- `"default"` — текущая пара, `Manrope` (текст) + `Space Grotesk` (заголовки).
- `"alt"` — новая пара, `Inter` (текст) + `Sora` (заголовки). Обе гарнитуры
  добавляются в тот же `<link>` в `__root.tsx` (расширение существующего
  URL параметра `family=`), так что обе пары всегда предзагружены и
  переключение — это просто смена значения CSS-переменных
  `--font-sans`/`--font-display`, без динамической загрузки шрифтов.

`frontend/src/lib/theme-manager.ts` получает новую функцию
`applyFontPreset(id: "default" | "alt")`, устанавливающую
`--font-sans`/`--font-display` на `document.documentElement`, по аналогии
с `applyAccentPreset`.

### 6. No-JSON UI

Два `<select>` (2 опции цвета, 2 опции шрифта) + кнопка «Опубликовать для
всех» + условно видимая кнопка «Откатить последнее изменение». Никаких
текстовых полей, никакого сырого JSON — значения строго из фиксированных
enum на фронте, отправляются как структурированный объект, не
редактируемая строка.

## Данные не меняются на бэкенде, кроме одного нового эндпоинта

- Новое: `GET /design-system` (публичный, без авторизации) —
  `PublicContent`-модуль, по образцу `FeatureFlagsController`.
- Существующее, без изменений: `SystemSetting` модель, `PUT /admin/settings`,
  `AuditService::log()`, `GET /admin/audit-logs`.
- Никаких новых миграций.

## Тестирование / приёмка

- `tsc --noEmit` чист.
- Live-проверка на demo-стенде:
  - Админ меняет цвет в разделе Design System → видит превью только у себя
    (другая вкладка/инкогнито того же сайта под другим пользователем не
    видит изменения до публикации).
  - Нажатие «Опубликовать для всех» → `GET /design-system` отдаёт новое
    значение → у нового посетителя (свежая загрузка, не demo-режим)
    применяется опубликованный цвет/шрифт.
  - Появляется новая запись в audit log (раздел «История изменений» из
    фазы 1) с `action = 'admin.settings.update'`.
  - «Откатить последнее изменение» возвращает предыдущее опубликованное
    значение, создаёт свою запись в audit log.
  - Если audit log пуст (design_system никогда не публиковался) — кнопка
    «Откатить» не отображается.
- Overflow-пробы 360/390/430 на обновлённом разделе Design System.

## Не входит в эту фазу

- Полная замена набора иконок (lucide → другой набор) — фаза 2b, отдельная
  спека, отдельный implementation plan (рефакторинг импортов по всему коду).
- Произвольный hex-цвет или кастомный шрифт для публикации — сознательно
  вне скоупа; advanced/debug hex-путь в `theme-manager.ts` остаётся
  доступным как раньше (для локального дебага), но не подключается к
  публичному `PUT /admin/settings`.
- Полная история версий design system с возможностью выбрать произвольную
  точку отката (не только последнюю) — если понадобится, админ уже может
  прочитать любую историческую запись `old_values` вручную в разделе
  «История изменений» (фаза 1); отдельный UI для «откатиться на N шагов
  назад» не строится сейчас.
